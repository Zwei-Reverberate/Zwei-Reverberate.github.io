    C++ Concurrency

    1. 并发基础

    并发：指两个或两个以上的独立活动同时发生。计算机的并发通常指单个系统同时执行多个独立的任务。

    • 并发的两种方式：

      • 真正并行

        依赖硬件的多核或多处理器架构，每个处理器或核心都独立执行任务

      • 任务切换

        在单个处理器上快速切换执行多个任务，使得它们看起来像是同时进行

        操作系统通过分配给每个任务很短的执行时间片（time slice），并在这些时间片之间切换任务，以实现并发的效果。这也会造成时间和空间上的额外开销

    • 进程与线程

      • 进程

        进程是操作系统拥有资源基本单位。每个进程都有自己的独立地址空间，包括代码段、数据段、堆和栈。一个进程可以包含多个线程

      • 线程

        线程是进程中的一个执行单元，是独立调度的基本单位。一个进程可以包含多个线程，这些线程共享进程的地址空间和资源（如全局变量、文件描述符等）

      • 多进程并发

        多进程并发是将应用程序分为多个独立的进程同时运行，通过进程间的通讯渠道传递讯息（信号、套接字、文件、管道等）

        进程间通信复杂且缓慢。操作系统会对进程进行保护，以避免一个进程去修改另一个进程的数据

      • 多线程并发

        多线程并发是在单个进程中运行多个线程。由于线程间地址空间共享且缺乏数据保护，所以开销远远小于多进程

        线程并不是越多越好，每个线程都需要一个独立的堆栈空间（1M）[]，线程之间的切换要保存很多中间状态，会耗费本该程序运行的时间

    3. 共享数据

    1. 条件竞争

    条件竞争（Race Condition）是指多个线程在访问和操作共享数据时，由于线程间的执行顺序无法预测，导致程序行为不确定甚至出现错误的情况。这种问题通常发生在以下场景：

    • 多个线程同时读取和写入同一块内存区域
    • 一个线程在写入共享数据的同时，另一个线程正在读取该数据
    • 缺乏适当的同步机制来控制对共享资源的访问

    2. 互斥量

    解决条件竞争的一种方法是使用互斥量，简单来说就是访问共享数据前，将数据锁住，访问结束后，再将数据解锁。当线程使用互斥量锁住共享数据时，其他线程都必须等到之前那个线程对数据进行解锁后，才能访问数据。如果其他线程此时试图访问共享数据，就会陷入阻塞，直至数据被解锁。

    3. 死锁

    死锁（Deadlock）是多线程编程中的一个严重问题，它指的是两个或多个线程相互等待对方释放资源，从而导致程序陷入僵局，无法继续执行。

    • 死锁的条件

      1. 互斥条件（Mutual Exclusion）：资源不能同时被多个线程占用
      2. 保持和等待条件（Hold and Wait）：一个线程至少已经持有一个资源，并且还在等待获得其他被其他线程持有的资源
      3. 不剥夺条件（No Preemption）：资源不能被强制从线程中剥夺，只能由持有线程主动释放
      4. 环路等待条件（Circular Wait）：存在一个线程等待环，其中的每个线程都在等待一个被下一个线程持有的资源

      3. 避免嵌套锁

        尽量避免在线程持有一个锁的情况下再去获取另一个锁，尽量使每个线程只持有一个锁

      4. 避免在持有锁时调用外部代码

        外部代码包括任何你无法完全控制的代码，例如库函数、回调函数、虚函数（可能在派生类中重载）以及任何可能执行不确定操作的代码。持有锁时调用这些代码可能导致难以预测的行为和死锁，因为你无法控制这些代码内部是否会再次尝试获取锁或者执行其他阻塞操作。

      5. 使用层次锁结构

        使用层次锁结构是一种编程实践，用于避免死锁问题。通过对锁的获取顺序进行严格约束，确保所有线程在获取多个锁时遵循同样的顺序，从而避免循环等待的情况。

        层次锁结构的核心思想是为每个锁分配一个层级（层次），并确保所有线程按照锁的层级顺序获取锁：

        • 每个锁被分配一个唯一的层级号（通常是一个整数）
        • 线程在持有较低层级的锁时，只能获取较高层级的锁，不能反过来

        通过这种方式，可以防止循环等待的情况，因为锁只能按照严格的层级顺序获取，从而避免了可能的死锁

    • std::unique_lock

      std::unique_lock 是 C++11 中引入的一种灵活的锁管理器，用于管理互斥量的所有权。

      它相比于 std::lock_guard 提供了更多的功能和更大的灵活性，比如支持延迟锁定、尝试锁定、超时锁定、手动解锁和重新锁定等操作。

      使用场景：

      1. 延迟锁定（Deferred Locking）

        可以创建一个不立即锁定的 std::unique_lock 对象，然后在需要时再锁定。这在某些情况下可以提高程序的性能和灵活性

      2. 尝试锁定（Try Locking）

        可以尝试锁定互斥量，如果未能成功获取锁，不会阻塞当前线程，而是返回一个表示锁定结果的布尔值

      3. 超时锁定（Timed Locking）

        可以在指定的时间内尝试锁定互斥量，如果未能成功获取锁，返回一个表示锁定结果的布尔值。这在需要等待但不希望无限期阻塞的情况下非常有用。

      4. 手动解锁和重新锁定（Manual Unlocking and Relocking）

        可以手动解锁和重新锁定互斥量，这在某些复杂的情况下非常有用，比如在一个函数中多次锁定和解锁互斥量

      读写锁（读者-作者锁）是一种同步机制，允许多个线程并发读取共享数据，但在写入数据时需要独占访问。

      • 读锁（共享锁）：多个线程可以同时持有读锁，从而并发读取共享数据
      • 写锁（独占锁）：当一个线程持有写锁时，其他线程不能持有任何锁（读锁或写锁），确保写操作的独占性

      在很多应用中，读操作远比写操作频繁。传统的互斥锁（如 std::mutex）在这种场景下并不高效，因为它会在读操作之间也进行同步，导致不必要的性能损失。

      • std::shared_mutex

          • 共享锁之间不会相互阻塞，但共享锁会阻塞独占锁
        2. 独占锁（写锁）
          • 只允许一个线程持有独占锁，可以独占地写入数据
          • 独占锁会阻塞共享锁和其他独占锁

    4. 同步操作

    除了保护线程间的共享数据，同步也是并发编程实践中不可或缺的部分。例如，在某个线程完成前，等待另一个线程完成。

    1. 条件变量

    当一个线程等待另一个线程完成时，可以持续检查共享的数据标志这一方式简单实现：

    • 使用 std::atomic<bool> 类型的变量 is_done 来表示工作线程是否完成任务，std::atomic 提供了原子操作，保证在多线程环境中不会出现数据竞争
    • 工作完成后，将 is_done 标志设置为 true，使用 std::memory_order_release 来确保内存顺序
    • 为避免忙等待，线程在每次检查后休眠一小段时间（100 毫秒）

    这样虽然能达到目的，但也有诸多弊端：

    • 忙等待：监视线程持续检查标志，并在每次检查后短暂休眠，这被称为忙等待。忙等待会导致 CPU 资源的浪费
    • 延迟：由于监视线程需要周期性地休眠和检查标志，可能会导致延迟。休眠时间也难以控制
    • 扩展性差，难以适应复杂的同步场景
    • 不适用于多核处理器：在多核处理器环境中，忙等待会导致多个核的无效使用，因为线程在忙等待期间会占用 CPU 时间片，而不是进行有用的工作

    为了克服这些弊端，可以使用 C++ 提供的条件变量。

    条件变量（Condition Variable）是多线程编程中的一种同步机制，用于在线程之间实现复杂的等待 / 通知模式。条件变量允许一个或多个线程等待某个条件变为真，然后再继续执行。

    主要作用：

    1. 等待（Wait）：线程可以等待某个条件变量，直到某个条件变为真
    2. 通知（Notify）：当某个线程修改了共享数据并使条件满足时，可以通知等待该条件的线程继续执行

    std::condition_variable 和 std::condition_variable_any 是两种用于线程同步的条件变量：

    • std::condition_variable

      std::condition_variable 是标准库中的条件变量，通常与 std::unique_lock<std::mutex> 一起使用。它只能与 std::mutex 一起使用，用于在同一程序中进行线程同步。

      主要成员函数：

      • wait(std::unique_lock<std::mutex>& lock): 使当前线程阻塞，直到被其他线程通知或被虚假唤醒。该函数会在阻塞前自动解锁 lock，在被唤醒后自动重新加锁
      • wait(std::unique_lock<std::mutex>& lock, Predicate pred): 阻塞当前线程，直到被通知并且 pred 返回 true
      • notify_one(): 通知一个等待该条件变量的线程
      • notify_all(): 通知所有等待该条件变量的线程

      std::condition_variable_any 是更通用的条件变量，可以与任何类型的锁（包括用户定义的锁类型）一起使用，而不仅仅是 std::mutex

    2. 使用 future

    std::future 是 C++11 标准库中的一个模板类，用于表示异步操作的结果。提供了一种在任务完成后获取结果的机制，无需忙等待。

    std::future 通常与 std::async、std::promise 和 std::packaged_task 一起使用。

    假设有一个需要长时间计算的任务，这个任务的结果是一个有效值。可以启动一个新线程来执行这个任务，而 std::thread 会忽略返回值。这时候可以通过 std::async 解决。

    • std::async

      std::async 是一种启动异步任务的高层方法，并返回一个 std::future 对象

      std::async 的行为取决于其第一个参数 std::launch 的标志：

      2. std::launch::deferred

        不会启动新线程，任务的执行被推迟，直到 future.get() 或 future.wait() 被调用时在调用线程中执行

    • std::packaged_task

      std::packaged_task 是 C++11 引入的一个模板类，用于包装一个可调用对象（如函数、lambda 表达式、函数对象等），并将其结果存储在 std::future 对象中

      主要用途

      • 任务包装：将一个可调用对象包装成一个任务，以便异步执行
      • 结果存储：将任务的执行结果存储在 std::future 对象中，以便异步获取

      使用方法

      3. 启动任务：通过调用 std::packaged_task 对象，启动任务的执行
      4. 获取结果：通过 std::future 对象的 get() 方法获取任务的执行结果

    • std::promise

    5. 线程池

    线程池是一种常用的并发设计模式，用于管理和复用一组预先创建的线程来执行任务。

    线程池的目的是减少线程创建和销毁的开销，控制并发任务的数量，并提高系统的响应速度和吞吐量。

    1. 基本概念

    • 任务：需要在线程中执行的工作单元，可以是一个函数、一个计算单元、或任何可执行的操作
    • 工作队列：用于存储等待执行的任务。任务提交给线程池后，进入工作队列，等待线程池中的线程来执行
    • 工作线程：线程池中的线程，用于从工作队列中取出任务并执行
    • 线程池管理器：负责管理线程的创建、销毁、任务调度等

    2. 优点

    • 减少资源消耗：通过复用线程，避免了频繁创建和销毁线程带来的资源消耗

    • 提高响应速度：由于线程池中的线程通常是预先创建的，任务可以更快地开始执行

    • 控制并发数量：可以通过配置线程池的大小来限制同时运行的线程数，避免系统资源被过度消耗

    • 简化管理：统一管理线程的创建、销毁和任务调度，简化了多线程编程的复杂性。

    3. 示例

    关键点：

    1. 线程创建和管理：线程池在构造函数中创建一组线程，这些线程在析构函数中被安全地销毁
    2. 任务队列：使用一个 std::queue 存储待执行的任务，并使用 std::function 来存储任务对象
    3. 互斥锁和条件变量：使用 std::mutex 保护任务队列，使用 std::condition_variable 来同步任务的提交和执行
    4. 任务提交：使用 enqueue 方法向线程池提交任务，该方法返回一个 std::future 对象，用于获取任务的结果
    5. 线程工作函数：每个线程在一个无限循环中运行，从任务队列中取出任务并执行

    可以改进的特性：

    • 动态调整线程数：可以实现动态调整线程池中的线程数，以适应不同的负载情况
    • 任务优先级：可以实现任务优先级，使得高优先级的任务优先执行
    • 任务超时：可以实现任务的超时处理，以防止某些任务长时间占用线程
    • 线程复用和销毁：可以实现线程的复用和销毁，以更好地管理资源
