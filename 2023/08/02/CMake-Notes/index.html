    CMake Notes

    1. CMake 简介

    1. 工程构建

    构建 (Build) 是程序开发的步骤之一。对于一个项目来说，其中可能包含不同的组件和模块，引用了各种第三方库，如果手动进行编译连接效率是比较低下的。

    所以我们一般会使用工程构建的方式全自动完成项目的编译，链接和打包。

    一些常见的 C++ 构建工具：GUN Make，CMake，MSBuild，Scons，Waf 等。

    2. CMake

    CMake 是一个开源的跨平台的 C++ 构建工具。

    CMake 自身不带有编译工具，使用 CMake 需要使用其他的如 MSVC，GCC，Clang 等编译工具。

    3. CMake 变量

    CMake 中变量是 ${变量名} 的形式。

    如 project(general) 命令自动生成了 PROJECT_NAME 这个变量，PROJECT_NAME是变量名，${PROJECT_NAME}是变量值，值为 general。

    又如 CMAKE_BINARY_DIR 是 CMake 的内置变量，代表 CMake 内部定义的构建目录，所有的二进制文件都在这个文件夹中产生。

    我们可以使用 message 命令打印出我们需要的信息，例如可以输出我们的 build 目录：

    CMake 常见的内置变量包括：

    变量名 说明
    CMAKE_SOURCE_DIR 源代码根目录，所有工程顶层目录
    CMAKE_CURRENT_SOURCE_DIR 当前处理的 CMakeLists.txt 所在的路径
    PROJECT_SOURCE_DIR 当前 project 顶层目录
    PROJECT_BINARY_DIR 工程构建目录
    PROJECT_IS_TOP_LEVEL BOOL 类型，表示当前项目是否是根项目
    CMAKE_BINARY_DIR 根项目输出路径

    2. 库（library）

    有时我们会有多个可执行文件，它们之间的某些功能是相同的，于是这些公共的功能就可以做成一个库，以便于共享。

    一个库中的函数既可以被可执行文件调用，也可以被其他库文件调用。

    库文件分为静态库文件和动态库文件：

    • 静态库文件

      相当于直接把代码插入到生成的可执行文件中，是编译时查找，会导致可执行文件体积变大

    • 动态库文件

      运行时查找。只在可执行文件中生成“插桩”函数，当可执行文件被加载时会读取指定目录的 .dll 文件，加载到内存中的空闲位置，并且替换相应的“插桩”指向的地址为加载后的地址，这个过程称为重定向。

    2. 项目的配置变量

    1. CMAKE_BUILD_TYPE

    CMAKE_BUILD_TYPE 是 CMake 中一个特殊的变量，用于控制构建类型，它有以下选项：

    • Debug：调试模式，完全不优化，生成调试信息，方便调试程序
    • Release：发布模式，优化程度最高，性能最佳，但是编译比 Debug 慢
    • MinSizeRel：最小体积发布，生成的文件比 Release 更小，不完全优化，减少二进制的体积
    • RelWithDebInfo：带调试信息发布，生成的文件比 Release 更大，因为带有调试符号信息
    • 默认情况下 CMAKE_BUILD_TYPE 胃空字符串，这时相当于 Debug

    project() 命令中有一个名为 LANGUAGES 的字段，指定项目使用了哪些编程语言。

    当然我们也可以不设置，而是在之后调用 enable_language() 命令也是一样的效果。

    add_library 无参数时，会根据 BUILD_SHARED_LIBS 这个变量的值来决定是静态库还是动态库。ON 代表选择 SHARED, OFF 代表选择 STATIC。如果未指定 BUILD_SHARED_LIBS 这个变量的值，则默认为 STATIC。

    动态库无法链接静态库

    6. 变量与作用域

    变量传播规则：

    • 父模块里定义的变量，会传播给子模块

    • 子模块中的变量不会传播给父模块

    • 父模块与子模块定义了同名变量，在子模块中执行子模块的定义，离开子模块后仍保持父模块定义

    • 如果子模块想要向父模块中传递变量，可以使用 PARENT_SCOPE 选项，把变量传递给上一层作用域（也就是父模块）:

    • 除了父子模块外，function 也有独立作用域，而 macro 没有独立作用域

    环境变量、缓存变量：

    使用 $ENV{XXX} 可以访问环境变量，使用 $CACHE{xxx} 可以访问缓存变量，二者是全局的，不分作用域。

    使用 if(DEFINED XXX) 可判断某变量是否存在

    7. include

    我们可以编写独立的 .cmake 文件，其作用类似于脚本。可以使用 include 命令将其包含进 CMakeLists.txt 文件。其中定义的各种函数和指定和在 CMakeLists.txt 中书写没有区别。

    Include 和 add_subdirectory 的区别：

    • include 相当于直接把代码粘贴过去，直接访问调用者的作用域。这里创建的变量和外面的变量共享，直接 set(key val) 则调用者也有 ${key} 这个变量了
    • add_subdirectory 中则给予定义者所在路径，优先访问定义者的作用域。需要 set(key val PARENT_SCOPE) 才能修改到外面的变量

    5. 第三方库

    当我们需要为项目引入第三方库时，CMake 针对不同的需求提供了几种不同的方法：

    • 使用已安装的库

      如果第三方库已经被安装在了系统中，CMake 提供了 find_package 命令用于在系统中查找已经安装的库。这通常涉及到使用 Find(PackageName).cmake 模块，该模块描述了如何查找和配置指定的库。

    • 使用 ExternalProject_Add

      如果项目依赖一个外部库，但不想事先将其安装在系统中，可以使用 ExternalProject_Add 来在构建时下载、编译和构建该库。这样，项目就可以在构建时自动获取和构建依赖。

    • 手动配置和构建

      如果项目包含一个源码树中的库，你可以使用 add_subdirectory 来添加并构建这个库。这种方法适用于你有库的源码而不是预编译二进制文件的情况。比如某些作为头文件引入的第三方库。

    • 使用 FetchContent

      类似于ExternalProject_Add，但更加直观。需要 CMake 3.11 及其以上版本

    1. find_package

    1. 用法举例

    • find_package(OpenCV)

      查找名为 OpenCV 的包，找不到不报错，事后可以通过 ${OpenCV_FOUND} 查询是否找到。

    • find_package(OpenCV QUIET)

      查找名为 OpenCV 的包，找不到不报错，不打印任何信息

    • find_package(OpenCV REQUIRED)

      常见用法，查找名为 OpenCV 的包，找不到报错，并终止 CMake 进程，不在继续往下执行

    • find_package(OpenCV REQUIRED COMPONENTS core videoio)

      查找名为 OpenCV 的包，找不到就报错，且必须具有 OpenCV::core 和 OpenCV::videoio 这两个组件，如果没有这两个组件也会报错

    • find_package(OpenCV REQUIRED OPTIONAL_COMPONENTS core videoio)

      查找名为 OpenCV 的包，找不到就报错。但没有对应的组件不会报错，可以通过 ${OpenCV_core_FOUND} 查询是否找到 core 组件

    2. 命令解释

    • find_package 实际上是在找对应的 xxxConfig.cmake 的文件，而不是去找具体的库文件和头文件
    • xxxConfig.cmake 配置文件中包含了包的具体信息，包括库文件的位置，头文件目录，链接时需要开启的编译选项等等
